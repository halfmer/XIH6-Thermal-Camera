而是给 README_2 ~ README_14 这十三篇战报做一次全景收束：系统最终停在

## 1. 最终形态

```text
FLIR LEPTON 3.5 ──VoSPI(SPI4)──> STM32H743XIH6 ──┬─ UART4/CH340C @1.5Mbps ──> Qt 上位机(串口模式)
                                  │              └─ SPI5 从机 ──> ESP32-S3 ──TCP──> Qt(WiFi) / Android
                                  │
                 SHT40 · MQ-2 · MQ-135 · OLED · SD · 火灾LED(PJ15) · 蜂鸣器(PG9)
                                  └─ 协作时间轮伪同时调度（视频永远第一优先客户）
```

| 端 | 收卷时状态 |
|---|---|
| STM32 固件 | 基线 1b19dfd（HEX MD5 **160CD0F1**）：~2FPS 完整帧流 + 时间轮多外设 + MQ ppm 火灾预警。工作区另有 §9 过温报警改动（代码完成，**未构建未上板**） |
| Qt 上位机 | 串口/WiFi 双模式收流，checksum 校验 + 撕裂帧段缓存重组 + 百分位色图，只显示完整帧 |
| ESP32-S3 桥 | SPI 主机拉流 + TCP 转发已通，DRDY 握手 + 500ms 超时自愈 |
| Android | esp32_demo v1.0 编译通过（AA55 协议与 Qt 对齐；安卓当 TCP 服务器 8888），端到端联调未完成 |
| 硬件遗留 | CCI(I2C4) 5 轮排查未通（硬件问题搁置）——LEPTON 全程跑出厂默认配置，TLinear 恰好默认开启救了测温 |

版本锚点链：`C87FBCC8`(2FPS金标准) → `567EF0AA`(双链路+桥+时间轮) →
`6091ACF1`(MQ135 双通道) → **`160CD0F1`(收卷基线)**；
组合改动 `f9b07240`(2864b27) 上板更差 → 整体 revert(dbb4133)，是单变量原则的判例。

## 3. 达成 vs 未竟（对照 AGENTS.md 成功定义）

```text
✅ LEPTON 持续出流            ~2FPS，长时间运行稳定
✅ STM32 只发完整帧           staging 缓冲 + 同帧 checksum，未见半帧出线
✅ ESP32 可靠转发             DRDY 握手 + 超时自愈
✅ Qt 只显示完整帧/零撕裂     checksum 门禁 + 段缓存重组后未再复现撕裂
✅ 帧完整性 > 帧率            全程贯彻（宁要 2FPS 完整帧）
➖ Android 行为与 Qt 一致     协议层对齐、客户端编译通过，端到端未联调
❌ 8~12 FPS                  停在 ~2FPS：采集节奏为第一瓶颈，CH340C 为链路瓶颈
❌ 24h 长稳                  未及测试
❌ CCI 控制通道               硬件搁置——FFC/增益/AGC 永远只能出厂默认
❌ 过温报警上板               代码与文档完成（README_14 §9），构建与验证未跑
```

## 4. 技术遗产——如果重来一次

1. **先量硬件，再改软件。** 本项目四个最大的坑——GPIO 复用根因、CH340C FIFO
   丢字节、ADC 1.5 周期采样充不满、栈溢出——没有一个是算法问题。示波器和
   逻辑仪比重构代码便宜。
2. **单变量上板。** 2864b27 五合一"组合革命"上板更差且无法归因，只能整体
   revert。之后每一项改动独立构建、独立验证、独立提交。
3. **留金标准锚点。** "2FPS 金标准 C87FBCC8"让每次激进尝试都有底可退，
   回退不是失败，是保住已验证的事实。
4. **热路径零打印（Step9A 红线）。** DMA 飞行中的一条阻塞打印能以"每第 16 帧
   坏一帧"这种最难查的方式还债。
5. **HEX MD5 写进 commit message。** "板上跑的到底是哪一版"这个千古之问，
   永远有答案。
6. **运行时重配置防 CubeMX regen。** Lepton_SPI_Config / Stream_Init / ADC
   810.5 周期全用同一模式：Cube 生成什么无所谓，跑起来之前我再改一遍。
7. **时间轮的正确姿势：视频不进轮。** README_10 失败与 README_14 成功之间
   只差这一条——第一优先客户无条件先跑，轮子只管背景任务。
8. **fail-loud。** 火警不做超时静音：没有新数据时维持报警，直到画面亲自变凉。

## 5. 交接点（若有后来者，含未来的自己）

1. **固件基线**：checkout `dbb4133`（= 160CD0F1 内容）即为最后上板验证过的
   版本；工作区的过温报警改动（fire_guard.c/h + main.c + README_14 §9）逻辑
   完整但从未编译——接手先构建 0E/0W 再上板。
2. **提帧率路线**（按性价比排序）：VoSPI discard 包处理节奏 → 换 CH343P
   （1.5KB FIFO，可回 2Mbps+）→ SHT40 等慢外设已非阻塞化，不再是障碍。
3. **CCI**：查 I2C4 上拉与电平匹配；通了就能做 FFC 手动触发与增益切换，
   测温精度和画质上限都在这条路上。
4. **Android**：ESP32 侧写死 `192.168.1.100:8888`（部署 IP 坑，见
   esp32_demo 记录）；联调时先对这个。
5. **未提交工作区**：Android 客户端全套源码、AGENTS.md PowerShell 规范、
   AI_API_CONNECT 工具——收卷后建议做一次封箱 commit，避免散佚。
